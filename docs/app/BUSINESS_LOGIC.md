# Medilite AI Business Logic

This is the canonical product/business-logic file for the app. If a change affects user-visible workflow, permissions, statuses, or clinic operations, update this file before commit.

## Product Summary

Medilite AI is a clinic workflow system for Indian OPD clinics. It is designed for small to mid-sized clinics that need one system for intake, queue, consultation notes, billing/discharge, reports, and patient access.

The core promise is operational continuity:

1. patient identified or created
2. token created for the clinic/day
3. queue updated
4. consultation documented
5. billing/discharge completed
6. patient can later access continuity data

## Primary Actors

### Admin
- Can access staff-only clinic routes.
- Can view admin dashboard and settings.
- Can update doctor settings.

### Doctor
- Can access staff-only clinic routes.
- Can consult patients, generate SOAP drafts, complete visits, and view admin dashboard.
- Can update doctor settings.

### Receptionist
- Can access staff-only operational routes such as intake, queue, and patient list.
- Cannot access admin/settings surfaces unless explicitly allowed by code.
- Can create appointments and move patients through front-desk workflow.

### Patient
- Is not a regular staff-authenticated user.
- Can access clinic public routes such as booking and portal flows.
- Can access portal through OTP login or a secure time-limited magic link.

## Tenant Model

- Every clinic has a `slug`.
- Staff routes are under `/{slug}/...`.
- Clinic can also resolve through custom domain.
- All clinic-scoped data must be isolated by `clinic_id`.

## Staff Authentication Model

- Staff auth uses HMAC-signed cookie `swasthya_session`.
- `src/proxy.ts` resolves the clinic and injects:
  - `x-clinic-id`
  - `x-user-id`
  - `x-user-role`
- Protected clinic routes require a valid session belonging to that clinic.

## Patient Access Model

There are two patient access paths:

### OTP Portal Login
- Route: `/{slug}/portal/login`
- Patient enters phone number.
- OTP is sent on WhatsApp.
- If OTP verifies, patient gets a patient session cookie.
- If multiple profiles share the number, patient selects the correct profile before entering the portal.

### Magic-Link Portal Access
- Route: `/{slug}/portal/[token]`
- Token is validated against `patient_access_tokens`.
- This is a secure, time-limited access path generated by the clinic/system.

### Legacy Token View
- Route: `/{slug}/patient/[token]`
- Legacy patient-facing token route still exists.
- Treat as legacy/public continuity surface, not the primary future-facing portal model.

## Appointment Lifecycle

Current appointment statuses:

- `booked`
  Created but not yet marked as arrived.

- `confirmed`
  Patient is present and in the queue.

- `in_progress`
  Doctor is currently consulting the patient.

- `completed`
  Consultation is done.

- `cancelled`
  Visit explicitly cancelled.

- `no_show`
  Patient did not arrive.

- `rescheduled`
  Visit moved to a different slot/date.

## Core Workflow Rules

### Clinic Onboarding
- Route: `/onboard`
- Creates a clinic plus default doctor profile.
- Slug must be unique.
- This is clinic creation, not patient registration.

### Reception Intake
- Main staff route: `/{slug}/intake`
- Receptionist can search by phone/name, reuse an existing patient, or create a new patient.
- Creating an appointment generates a daily clinic token number through the database.
- Reception intake typically places the patient into `confirmed` because they are present in the clinic.

### Public Booking
- Public route: `/{slug}/book`
- Patient can book without staff login.
- Bookings enter the clinic workflow for that date and clinic.
- This is public intake, not full patient self-service.

### Queue
- Main route: `/{slug}/queue`
- Queue is the clinic‚Äôs day-of-operation view.
- Main active statuses in the queue are `booked`, `confirmed`, `in_progress`, and `completed`.
- ‚ÄúCall next patient‚Äù and status changes move the operational flow forward.

### Consultation
- Main route: `/{slug}/queue/[appointmentId]/consult`
- Doctor can document consultation manually.
- AI SOAP drafting is assistive, not authoritative.
- Final saved visit record is the truth, not the draft.

### Visit History
- Completed consultation data is stored in `visit_history`.
- This history is used for continuity in patient and portal views.

### Reports
- Patient reports can be uploaded and attached to the patient record.
- Parsing is optional and AI-assisted.
- Parsed output is a convenience layer; the uploaded report remains the source artifact.

### Billing And Discharge
- Billing data lives in `bills`, `bill_line_items`, and `payment_events`.
- Appointment-level payment snapshot is derived from billing/payment state.
- Billing can be partial, mixed-mode, paid, pending, failed, etc.
- Discharge should reflect actual consultation/billing completion, not just UI clicks.

### WhatsApp
- Outbound patient messages go through `src/lib/whatsappAdapter.ts`.
- Communication events should be logged.
- WhatsApp is an operational channel, not a replacement for core clinic state.

## Doctor Settings Rules

- Main route: `/{slug}/settings`
- Only doctor/admin roles should be allowed to use this page and update settings.
- Settings include:
  - doctor name
  - speciality
  - phone
  - working hours
  - slot duration

## Admin Dashboard Rules

- Main route: `/{slug}/admin`
- This is a staff insight/oversight surface, not the core OPD workflow.
- It should be limited to doctor/admin roles.
- Dashboard metrics must be derived from real appointments/audit data, not made up.

## AI Rules

- Sarvam AI is used for voice and extraction features.
- AI can draft or structure information.
- AI must not silently invent missing medical or patient facts.
- Manual clinic flow must still work if AI fails.

## Explicit Non-Rules

These things should not be inferred unless the code explicitly supports them:

- enterprise EMR integrations
- automated clinical decision making
- patient self-service beyond booking and portal access
- changing the global visual theme without explicit user request
